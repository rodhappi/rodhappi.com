# Specification: GitHub Pages Deployment

**Feature:** Deploy www.rodhappi.com to GitHub Pages
**Status:** Draft
**Created:** 2026-03-30

---

## 1. Problem Statement

The site is fully built (static HTML/CSS/JS) but has no way to go live. The user needs the simplest, cheapest path to put `www.rodhappi.com` online.

**Why GitHub Pages:** Free, integrates with the existing GitHub repo (`rodhappi/rodhappi.com`), supports custom domains with HTTPS, and requires minimal configuration. No account setup beyond what already exists.

## 2. User Stories

### US-1: Automated Deploy on Push
**As** the site author,
**I want** the site to automatically build and deploy when I push to `main`,
**So that** I don't have to manually run build steps or upload files.

**Acceptance Criteria:**
- Pushing to `main` triggers `node build.js` and deploys the output
- The live site reflects the latest `main` within 5 minutes (typical: ~2 min)
- Failed builds do not take down the live site

### US-2: Custom Domain
**As** the site author,
**I want** the site accessible at `www.rodhappi.com`,
**So that** visitors use my personal domain, not `rodhappi.github.io`.

**Acceptance Criteria:**
- Site is reachable at `www.rodhappi.com`
- HTTPS is enabled (GitHub Pages provides this free via Let's Encrypt)
- `rodhappi.github.io` redirects to the custom domain

### US-3: Reliable Serving
**As** a visitor,
**I want** the site to load fast and reliably,
**So that** I have a good reading experience.

**Acceptance Criteria:**
- Static assets served from GitHub's CDN
- No server-side dependencies or runtime errors possible
- All pages return HTTP 200 and render correctly

## 3. Functional Requirements

| ID | Requirement |
|----|-------------|
| FR-1 | GitHub Actions workflow runs `node build.js` on push to `main` |
| FR-2 | Workflow deploys generated files (`index.html`, `posts/`, `css/`, `js/`, `media/`) to GitHub Pages |
| FR-3 | `CNAME` file with `www.rodhappi.com` is included in deployed output |
| FR-4 | Build failures prevent deployment (no broken deploys) |

## 4. Non-Functional Requirements

| ID | Requirement |
|----|-------------|
| NFR-1 | Deploy completes in under 3 minutes |
| NFR-2 | Zero cost (GitHub Pages free tier) |
| NFR-3 | HTTPS enforced on custom domain |
| NFR-4 | No secrets or tokens stored in the repository (use `GITHUB_TOKEN` provided by Actions) |

## 5. Technical Approach

### Build Output

`build.js` generates files **in the project root** (no separate output directory):
- `index.html` — generated in root
- `posts/*.html` — generated in `posts/`

These coexist with source files. The workflow must select which files to deploy.

### GitHub Actions Workflow (`.github/workflows/deploy.yml`)

1. **Trigger:** `push` to `main` branch
2. **Steps:**
   - Checkout repo
   - Setup Node.js (version 20, current LTS)
   - Run `node build.js`
   - Create a `CNAME` file containing `www.rodhappi.com` in the project root (generated by the workflow, not committed to the repo)
   - Use `actions/upload-pages-artifact` to package only the deployable files: `index.html`, `posts/`, `css/`, `js/`, `media/`, `CNAME`
   - Use `actions/deploy-pages` to publish
3. **Permissions:** `contents: read`, `pages: write`, `id-token: write`

### CNAME Configuration

- The `CNAME` file is created by the workflow during build (not committed to the repo)
- User must configure DNS at their domain registrar (see Section 9)

### What Gets Deployed

Only the files needed to serve the site:
- `index.html` (generated by build)
- `posts/*.html` (generated by build)
- `css/style.css`
- `js/nav.js`
- `media/*` (all media assets)
- `CNAME` (generated by workflow)

**Not deployed:** `content/`, `templates/`, `build.js`, `specs/`, `knowledge/`, `.claude/`, `appendix/`, `.git/`, `node_modules/`

## 6. Edge Cases

| Case | Handling |
|------|----------|
| Build fails (bad markdown) | `node build.js` exits non-zero, workflow stops before deploy step, previous GitHub Pages deployment remains live (atomic deploys) |
| No content files | Build produces empty index, deploys (not harmful) |
| Large media files | GitHub Pages has 1GB soft limit; current site is well under |
| Concurrent pushes | GitHub Actions queues workflows sequentially |

## 8. Out of Scope

- CDN or caching beyond what GitHub Pages provides
- Preview deployments for branches (can add later)
- Analytics or monitoring
- Email or other domain services
- Apex domain redirect (can configure later at registrar level)

## 9. DNS Setup (Manual Step)

The user needs to configure DNS at their domain registrar. This is outside the repo but documented here for completeness:

| Record Type | Name | Value |
|-------------|------|-------|
| CNAME | `www` | `rodhappi.github.io` |
| A (optional, for apex) | `@` | `185.199.108.153` |
| A (optional, for apex) | `@` | `185.199.109.153` |
| A (optional, for apex) | `@` | `185.199.110.153` |
| A (optional, for apex) | `@` | `185.199.111.153` |

After DNS propagation, enable "Enforce HTTPS" in the GitHub Pages settings.
