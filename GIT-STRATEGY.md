# Git Strategy

Git workflow aligned with the AI-Assisted Development Framework phases.

## Branching Model

- **`main`** — always deployable, represents what's live on `www.rodhappi.com`
- **Feature branches** — one per `/specify` → `/compound` workflow cycle

```
main
 └── feature/mobile-responsive
 └── feature/new-post-layout
```

Branch naming: `feature/<spec-slug>`, matching the spec folder name in `specs/`.

## When to Commit

| Phase | Commit? | What to capture |
|-------|---------|-----------------|
| `/specify` | Yes | The spec file in `specs/` |
| `/clarify` | Yes | Updated spec with resolved ambiguities |
| `/plan` | Yes | `plan.md` + `tasks.md` |
| `/approve` | No | Just a gate, no artifacts change |
| `/work` | Yes, per task | Each completed task = 1 commit |
| `/review` | Optional | If review findings cause fixes |
| `/compound` | Yes | New `knowledge/` entries |
| `/archive` | Yes | Merged delta specs |

## Workflow

1. **Start on `main`** for `/specify` and `/clarify` (spec work lives on main)
2. **Branch before `/work`** — once the plan is approved, create the feature branch
3. **Commit after each task** — small, atomic commits tied to task IDs
4. **Merge to `main` after `/review` passes** — quality gate ensures it's clean
5. **Run `node build.js` before merging** — verify generated output locally (it's gitignored)

## Commit Message Convention

```
<type>: <short description>

[optional body with task/spec reference]
```

### Types

| Type | Usage |
|------|-------|
| `feat` | New feature or functionality |
| `fix` | Bug fix |
| `refactor` | Code restructuring, no behavior change |
| `docs` | Documentation changes |
| `spec` | Specification or plan artifacts |
| `knowledge` | Knowledge base entries from `/compound` |
| `style` | CSS/design changes only |
| `chore` | Build script, config, tooling |

### Examples

```
feat: add mobile-first responsive layout

Task: specs/mobile-first-responsive/tasks.md#T1
```

```
spec: add escritos card layout specification

Spec: specs/escritos-card-layout/spec.md
```

```
knowledge: capture responsive design patterns from review

From: /compound after mobile-first-responsive cycle
```

## What's Gitignored

See `.gitignore` for the full list. Key decisions:

- **`index.html` and `posts/`** — generated by `node build.js`, source of truth is `content/*.md`
- **`.claude/memory/`** — per-user session state, not shared
- **`.DS_Store`, editor files** — OS/IDE noise

## Rollback

- Git history maps 1:1 to workflow phases, so rollback to any phase is straightforward
- Complements the framework's `/rewind` and `/checkpoint` skills
- Feature branches isolate experimental work from deployable `main`
